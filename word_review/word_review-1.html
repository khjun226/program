add: 깃 저장소의 작업트리에서 변경된 파일들을 스테이지 영역(index)에 추가하는 명령어입니다. 예를 들어, git add 파일명으로 특정 파일을, git add .로 모든 변경된 파일을 스테이지 영역에 추가할 수 있습니다. commit: 스테이지 영역에 있는 변경 내용을 기록하여 새로운 커밋을 만드는 명령어입니다. 예를 들어, git commit -m "커밋 메시지"를 사용하여 커밋 메시지와 함께 커밋을 생성할 수 있습니다. diff: 수정된 파일들의 차이를 보여주는 명령어입니다. 작업 디렉토리와 스테이지 영역의 차이를 보려면 git diff를, 스테이지 영역과 저장소의 차이를 보려면 git diff --staged를 사용합니다. pull: 원격 저장소의 변경 사항을 로컬 저장소로 가져오고 병합하는 명령어입니다. 예를 들어, git pull origin master를 사용하여 원격 저장소의 마스터 브랜치 변경 사항을 가져와 병합할 수 있습니다. fetch: 원격 저장소의 변경사항을 가져오지만, 자동 병합은 하지 않는 명령어입니다. 이후 수동으로 병합을 수행할 수 있습니다. push: 로컬 저장소의 변경 사항을 원격 저장소에 업로드하는 명령어입니다. 예를 들어, git push origin master를 사용하여 로컬의 마스터 브랜치를 원격 저장소의 마스터 브랜치에 푸시할 수 있습니다. reset: 특정 커밋으로 이동하거나 이력을 되돌리는 명령어 입니다. 예를 들어, git reset --hard 커밋ID를 사용하여 특정 커밋으로 이동하거나 작업 트리와 인덱스를 수정할 수 있습니다. restore: 작업트리의 파일이나 인덱스를 이전 상태로 되돌리는 명령어입니다. 예를 들어, git restore 파일명을 사용하여 특정 파일을 이전 커밋 상태로 되돌릴 수 있습니다. status: 작업트리의 상태를 확인하는 명령어입니다. 파일 변경, 스테이징 및 커밋되지 않은 파일 등의 정보를 표시합니다. log: 커밋 이력을 보여주는 명령어입니다. git log를 사용하여 커밋 이력을 확인할 수 있으며, 옵션으로 출력을 변경할 수 있습니다. CLI: CLI(Command-Line Interface)는 텍스트 기반의 명령어를 입력하여 컴퓨터 프로그램과 상호 작용하는 인터페이스입니다. 키보드 입력만으로 파일 관리, 프로세스 관리, 시스템 설정 변경 등의 작업을 수행할 수 있으며, 효율적이고 자동화에 유용합니다. 대표적으로 Unix/Linux 시스템의 Shell과 Windows의 명령 프롬프트가 있습니다. GUI: GUI(Graphical User Interface)는 그래픽 요소를 통해 프로그램과 상호 작용하는 인터페이스입니다. 직관적이고 사용하기 쉬워 일반 사용자에게 적합하며, 아이콘, 버튼, 메뉴 등의 시각적 요소로 구성되어 있습니다. 마우스와 키보드를 사용하여 작업을 수행하고, 동시에 여러 프로그램이나 작업 창을 열어 멀티태스킹을 할 수 있습니다. SSD: SSD는 Solid State Drive의 약자로, 하드디스크와 비슷한 역할을 수행하지만 내부 메모리에 반도체 칩을 사용하여 데이터를 저장하는 저장 장치입니다. HDD의 디스크 회전과 같은 물리적인 기계적 부품을 사용하지 않기 때문에 작동 속도가 빠르며 내구성이 높습니다. 그리고 전력 소모가 적어 노트북 등의 이동성 있는 기기에서도 활용할 수 있습니다. M.2: M.2는 새로운 저장장치 인터페이스 규격 중 하나로, PCI-E 버스를 사용하여 데이터 전송을 처리하는 저장 장치 입니다. M.2 슬롯에는 SSD나 외장 그래픽 카드 등이 설치됩니다. M.2를 이용한 저장장치는 SDD보다 더욱 높은 속도를 내며, 적은 전력으로도 높은 성능을 제공합니다. SSD와 M.2는 모두 내구성이 높고, 기계적인 구성 요소가 없어 방해요인이 적습니다. 또한, 저장장치로 사용할 때 더 신속하고 안정적인 작동이 가능합니다. 다만, M.2는 이전 버전의 인터페이스보다 필요한 보드가 더 작아져, 여러 모델 간 호환성이 있는지 확인해야 합니다. WOL: WOL(Wake on LAN)은 네트워크 상에서 컴퓨터를 원격으로 켜거나 깨울 수 있는 기술입니다. 이를 사용하면 다른 컴퓨터나 스마트폰, 태블릿 등의 장치를 통해 네트워크 상의 특정 컴퓨터를 켜거나 슬립 모드에서 깨울 수 있습니다. WOL은 일반적으로 BIOS 설정에서 활성화할 수 있으며, 네트워크상에서 패킷을 보내면 특정 MAC 주소를 가진 컴퓨터를 깨울 수 있습니다. 이를 활용하기 위해서는 WOL을 지원하는 프로그램을 별도로 설치해야 할 수도 있습니다. AC 부팅: AC 부팅은 전원 공급이 복원되면 컴퓨터가 자동으로 부팅되도록 하는 기능입니다. BIOS 설정에서 활성화 가능하며, 원격으로 시스템을 재시작할 때 유용합니다. 하지만 전원 공급이 중단되면 컴퓨터는 바로 꺼지고 데이터 손실 위험이 발생한다는 것이 AC 부팅의 단점이므로, 이 기능을 활성화하기 전에 중요한 데이터를 안전하게 백업하는 것이 좋습니다. 이러한 위험을 줄이기 위해, UPS(무정전 전원 공급 장치)를 사용하여 전원이 끊기더라도 잠시 동안 컴퓨터가 계속 작동하도록 할 수 있습니다. 쓰레드: 스레드(Thread)는 특정 프로그램 내에서, 특히 프로세스 내부에서 실행되는 흐름의 단위를 의미합니다. 일반적으로 하나의 프로그램은 단일 스레드를 포함하지만, 프로그램 환경에 따라 두 개 이상의 스레드를 동시에 실행할 수 있습니다. 이러한 실행 방식을 멀티스레딩(Multithreading)이라고 합니다. 멀티스레딩은 프로세스 내에서 여러 스레드를 동시에 실행하는 기법으로, 성능 향상, 자원 공유, 효율적인 작업 처리를 위해 사용됩니다. 이를 통해 프로그램 실행이 더 빠르고 경제적이며, 응답성이 높아집니다. 문맥 교환: 문맥 교환(Context Switching)은 컴퓨터 운영 체제의 프로세스 스케줄러가 실행 중인 프로세스나 스레드를 일시 중단하고, 다른 프로세스나 스레드를 실행하기 위해 현재의 상태와 환경을 보관하고 불러오는 과정입니다. 운영 체제는 멀티태스킹, 처리 우선 순위, 자원 할당, 프로세스 제어 등을 위해 문맥 교환을 수행하지만, 자주 발생하는 경우 시스템 오버헤드가 커져 성능이 저하될 수 있습니다. 따라서, 성능 최적화를 위해 문맥 교환 횟수를 최소화하는 것이 중요합니다. 데이터 경쟁상태: 데이터 경쟁 상태(Data Race 혹은 Race Condition)는 멀티스레드 환경에서 둘 이상의 스레드가 동시에 공유된 자원에 접근하고 변경 시도할 때 발생하는 문제입니다. 동시성 제어 조치를 취하지 않으면 예기치 않은 결과가 나타날 수 있습니다. 데이터 경쟁 상태가 발생하지 않도록 하는 방법으로는 상호 배제(Mutual Exclusion) 방식이 일반적으로 적용됩니다. 이 방식은 하나의 스레드만이 공유 자원에 접근하도록 보장하는 것입니다. 뮤텍스와 세마포어(Semaphore)와 같은 동기화 기법을 사용하여 이러한 상호 배제를 구현할 수 있습니다. 따라서, 데이터 경쟁 상태에 대한 이해와 적절한 대응이 필요하며, 다중 스레드 환경에서 프로그램을 개발할 때 항상 고려해야 할 중요한 사항입니다. REST-API: REST(API)는 Representational State Transfer(Aplication Programming Interface)의 약자로, 웹 API의 일종입니다. 여러 시스템 간의 HTTP 프로토콜을 사용하여 데이터를 주고받는 구조를 가지고 있습니다. REST API는 자원, 행위, 표현 과 같은 특징을 가지고 있으며 이러한 특징들은 REST API를 사용하여 다른 시스템과 데이터를 교환하고, 서로 상호작용할 수 있게 해줍니다. REST API는 최근에 많은 기업과 서비스에서 사용되고 있으며, 다양한 언어와 플랫폼에서 구현이 가능합니다. 또한, 클라이언트와 서버가 분리된 아키텍처를 사용하여, 서버 측과 클라이언트 측이 독립적으로 발전할 수 있는 이점을 가지고 있습니다. CI/CD: CI/CD는 소프트웨어 개발 프로세스에서 지속적인 통합(Continuous Integration)과 지속적인 배포(Continuous Deployment)의 약어입니다. 지속적인 통합(Continuous Integration) : 개발자들이 작성한 코드를 빌드, 테스트, 배포 자동화를 통해 일정한 주기로 통합하고 테스트하는 프로세스입니다. 코드 수정 후 빠르게 적용되며, 코드의 안정성과 품질을 유지하는데 도움을 줍니다. 지속적인 배포(Continuous Deployment) : 지속적인 테스트가 통과된 코드는 넘어가며, 최종적으로 사용자가 사용할 수 있도록 실제 서비스환경에 배치되는 프로세스입니다. CI/CD를 적용하면 개발자들은 개발한 코드를 더욱 빠르게 실제 환경에 적용할 수 있습니다. 코드 수정 후 매번 수동으로 통합, 빌드, 테스트, 배포 등의 일련의 작업을 거치지 않아도 자동화적으로 진행되므로, 버그나 잠재적인 문제 등을 미리 발견하고 대처할 수 있습니다. 애자일 기법: 애자일(Agile) 방법론은 소프트웨어 개발의 방법 중 하나로, 대규모 개발 프로젝트를 작은 단위로 쪼개어 짧은 주기로 개발하고 배포하는 방법입니다. 애자일 방법론은 일정한 기간 내에 적절한 기능을 제공하도록 시스템을 개발하고, 사용자의 요구사항을 수시로 반영하는 방법입니다. 애자일 방법론에서의 대표적인 프로세스로는 스크럼(Scrum)이 있습니다. 스크럼은 일정한 주기로 이루어지는 스프린트(Sprint)를 사용하여, 팀원의 역할과 결과물의 형식 등을 정의하고, 효율적인 프로젝트 관리를 할 수 있도록 합니다. 애자일 방법론의 주요 특징으로는 변화에 대한 대응, 작은 배포 주기 등이 있으며 이러한 특징을 통해 애자일 방법론을 사용하면, 요구사항의 변경, 빠른 제품 출시, 품질 향상 등의 이점을 얻을 수 있습니다. 또한, 변경사항에 대한 빠른 대응이 가능하고, 빠르게 결과물을 출시하며, 팀원들의 역량과 역할을 개선할 수 있습니다. 폭포수 기법: 폭포수 방법론 이란, 1970년에 창안된 소프트웨어 개발 프로세스입니다. 각 단계가 마치 폭포수처럼 위에서 아래로 물이 떨어지듯 차례대로 진행되어 이름이 붙여졌습니다. 여러 단계가 병행적으로 진행되거나 거꾸로 진행되는 경우는 거의 없고, 일정 생명주기를 반복하여 진행되는 애자일 방법과는 반대되는 개념입니다. 폭포수 모델의 장점으로는 요구사항 명확성, 변화에 대한 신속한 대처, 추적성 등이 있습니다. 또한 단계적인 진행 방식으로 인해 각 단계마다 산출물을 제공하여 전체적인 개발 진행 상황을 파악할 수 있습니다. 하지만 폭포수 모델의 단점으로는 요구사항의 불확실성, 높은 변경 비용 등이 있습니다. 또한 개발 과정 중간에 발생한 문제를 해결하기 어렵다는 단점이 있습니다. 이러한 이유로 최근에는 애자일(Agile) 방법론을 활용하는 경우가 많습니다. html-css-js: HTML(Hyper Text Markup Language), CSS(Cascading Style Sheets), JS(JavaScript)는 웹 개발에서 가장 기본이 되는 프론트엔드 기술입니다. HTML(Hyper Text Markup Language) : 웹 문서의 구조를 정의하는 언어로, 웹 페이지의 기본 뼈대를 제작합니다. 요소(Element)와 속성(Attribute) 등으로 이루어져 있습니다. CSS(Cascading Style Sheets) : 웹 문서의 스타일을 정의하는 언어로, HTML로 작성된 요소들의 디자인, 레이아웃 등을 담당합니다. 색상, 배경, 글꼴, 크기, 위치, 애니메이션 등을 지정할 수 있습니다. JS(JavaScript) : 웹 브라우저에서 동작하는 스크립트 언어로, HTML, CSS로 작성된 웹 페이지를 동적으로 만듭니다. 클라이언트 측 요소인 브라우저에서 동작하며, 이벤트 처리, 애니메이션, 유효성 검사, 즉시 실행 함수 등의 기능을 제공합니다. HTML, CSS, JS를 종합적으로 사용하여, 웹 사이트를 개발 및 디자인하며, 사용자 환경에 적합하게 작동하는 웹 어플리케이션을 개발할 수 있습니다. nginx: 가벼우면서도 강력한 프로그램을 목표로 러시아에서 개발되어 현재는 미국에서 운영중인 오픈 소스 웹 서버 프로그램이고 '엔진엑스'라고 읽습니다. nginx는 오픈소스 웹 서버, 리버스 프록시, IMAP/POP3 프록시 등 여러 가지 역할을 수행하는 서버 소프트웨어입니다. nginx는 사용하는 리눅스 서버에 대한 성능 및 확장성 향상, 보안성 향상 등을 위해 사용하는 경우가 많습니다. nginx는 적은 메모리와 기존의 웹 서버인 Aphache 보다 뛰어난 성능을 제공하며, 제한된 하드웨어 리소스로도 여러 클라이언트 요청을 동시에 효율적으로 처리할 수 있고, 단일 스레드를 통해 여러 연결을 처리 가능하는 등의 장점이 있습니다. 단점으로는 동적 컨텐츠를 기본적으로 처리 할 수 ​​없고, 동적 콘텐츠에 대한 PHP 및 기타 요청을 처리하려면 nginx가 이를 실행하기 위해 외부 프로세서로 전달하고 렌더링 된 콘텐츠가 다시 전송 될 때까지 기다려야 하는 등의 단점들이 있습니다. IP: IP는 인터넷 프로토콜(Internet Protocol)의 약어로, 인터넷이 통하는 네트워크에서 어떤 정보를 수신하고 송신하는 통신에 대한 규약을 의미합니다. IP 주소는 국제적으로 표준화된 주소 체계로써, 컴퓨터나 기타 장치들이 인터넷에 연결될 때 각기 다른 IP 주소를 가지게 됩니다. IP주소는 32비트 또는 128비트 숫자로 구성되며, 각 비트마다 0 또는 1의 값을 가집니다. IPv4는 32비트 주소 체계를 사용하고, 이를 통해 대략 42억 개의 IP주소를 사용할 수 있습니다. 그러나 이는 인터넷 확대에 따라 부족해졌고, IPv6의 등장으로 340경 개 이상의 IP 주소를 사용할 수 있게 되었습니다. IP 주소는 고유한 번호이기 때문에, 인터넷 상에서 컴퓨터 간 통신을 할 때는 상대방의 IP 주소를 이용하여 데이터를 송수신합니다. 이러한 IP 주소는 전 세계적으로 공유되고, 이를 통해 인터넷을 이용하여 전 세계의 컴퓨터와 통신할 수 있습니다. TCP와 UDP: TCP(Transmission Control Protocol)와 UDP(User Datagram Protocol)는 인터넷 프로토콜 스택에서 사용되는 두 가지 프로토콜입니다. TCP는 신뢰성 있는 데이터 전송을 위한 연결 지향적인 프로토콜로, 3-way handshake 라고 불리는 과정을 통해 연결 설정을 합니다. 데이터를 보내기 전에 먼저 연결을 설정한 후에, 데이터를 순서와 정확성을 유지하며 전송합니다. 그래서 TCP는 연속성보다 신뢰성있는 전송이 중요할 때에 사용하는 프로토콜로 예를 들면 파일 전송과 같은 경우에 사용됩니다. 또한, 전송되는 데이터의 크기에 따라 분할하여 전송하기 때문에, 대용량 데이터 전송에 적합합니다. UDP는 비연결성 프로토콜로, 포트 번호와 체크섬 정보만 있는 간단한 헤더 정보만으로 데이터를 전송합니다. TCP와 달리 연결 설정, 정확성 유지, 순서 보장 등의 기능이 없어, 신뢰성이 떨어질 수 있습니다. 그러나, 단순하고 빠른 속도로 대용량 빠른 전송이 필요한 경우, 신뢰성보다는 연속성이 중요한 서비스 예를 들면 실시간 서비스(streaming)에 자주 사용됩니다.
